HydraDB gives your AI three types of memory. Once you understand this model, everything else clicks.

## The Three Types of Memory

| Memory Type | What It Stores | Scope |
|-------------|----------------|-------|
| **User Memory** | Preferences, habits, past interactions | Per user (`sub_tenant_id`) |
| **Knowledge Memory** | Documents, files, app data | Shared or scoped to a sub-tenant |
| **Hive Memory** | Collective learnings across agents | Organization-wide |

## User Memory

**What your AI remembers about each person.**

User Memory captures preferences, communication style, past questions, likes, dislikes, and behavioral patterns. Each user is represented by a `sub_tenant_id`.

- **Ingest**: [`POST /memories/add_memory`](/api-reference/endpoint/add-memories)
- **Recall**: [`POST /recall/recall_preferences`](/api-reference/endpoint/recall-preferences)

**What you can build:**
- AI assistants that adapt to each user's style
- Support agents that remember past issues
- Apps that feel like they "know" the user

---

## Knowledge Memory

**What your AI knows from documents, apps, and data.**

Knowledge Memory is your AI's factual foundation вҖ” company policies, product docs, Slack threads, emails, Notion pages, and any unstructured data your agents need.

- **Ingest files**: [`POST /ingestion/upload_knowledge`](/api-reference/endpoint/add-knowledge-memories) (PDF, DOCX, TXT, etc.)
- **Ingest app sources**: same endpoint with `app_sources` form field (Gmail, Slack, Notion, etc.)
- **Recall**: [`POST /recall/full_recall`](/api-reference/endpoint/full-recall)

**What gets stored:**
- PDFs, docs, spreadsheets, presentations
- Slack messages, email threads
- Notion, Google Drive, Confluence pages

Every piece of knowledge is parsed, chunked, and indexed for recall.

---

## Hive Memory

**What all your agents learn together.**

Hive Memory is collective intelligence вҖ” learnings and institutional knowledge that every agent in your system can access.

- **Ingest**: [`POST /memories/add_memory`](/api-reference/endpoint/add-memories) (coming soon: dedicated endpoint)
- **Recall**: [`POST /recall/full_recall`](/api-reference/endpoint/full-recall)

---

## Tenants and Sub-Tenants

A **tenant** (`tenant_id`) is the top-level isolation boundary вҖ” one per customer organization or application.

A **sub-tenant** (`sub_tenant_id`) is a workspace within a tenant. Sub-tenants are created automatically on first use.

**B2B pattern**: Each customer organization is a separate `tenant_id`.
**B2C pattern**: One `tenant_id` for your app, one `sub_tenant_id` per user.

---

## Memory Recall

At query time, HydraDB retrieves the most relevant context using hybrid retrieval (semantic + keyword + graph traversal + personalized ranking).

| Endpoint | Searches |
|----------|----------|
| `POST /recall/full_recall` | Knowledge (documents, app sources) |
| `POST /recall/recall_preferences` | User memories |

### Recall Strategies

| Strategy | Best For |
|----------|----------|
| **Semantic** | Meaning-based search ("docs about refunds") |
| **Keyword** | Exact matches ("policy v2.3") |
| **Hybrid** | Best of both (default, recommended) |

---

## Metadata

Tag any memory with key-value metadata for deterministic filtering at recall time.

- **`metadata`** вҖ” tenant-level attributes set at ingestion (scoped by tenant schema)
- **`additional_metadata`** вҖ” document-level attributes set at ingestion
- **`metadata_filters`** вҖ” filter applied in recall requests before semantic ranking

**Common patterns:** `department`, `team`, `project`, `source`, `created_at`, `is_confidential`
